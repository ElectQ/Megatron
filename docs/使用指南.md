# Megatron 使用指南

个人情报雷达的中枢:**多源采集入库 → LLM 分级 → 门铃薄推送 + 日刊厚页面**。

想最快跑通一次,直接看 [快速上手](快速上手.md)。

## 1. 核心概念

### 配置目录 `config/` —— 我们的设计不焊在框架里

产品设计（源、提示词、任务、过滤策略）都在 `config/` 里，框架 `src/megatron/`
只留通用引擎。两种「真相」语义：

```
config/
  sources/*.yaml     数据源 —— 文件是唯一真相，UI 只读，改文件后 sync 覆盖 DB
  prompts/*.md       提示词 —— 文件是「种子」，首启导入；之后可在后台 UI 改，UI 赢
  tasks/*.yaml       分析任务 —— 同上（种子）
  policy.yaml        过滤策略默认：数量 caps + 政治黑名单（任务可逐个覆盖）
```

- `megatron profile validate` 校验 prompts/tasks；`megatron sources validate` 校验源。
- 首启（或 `megatron seed`）把 prompts/tasks **create-if-missing** 播种进 DB —— 已存在的不覆盖。
- 目录位置由 `MEGATRON_CONFIG_DIR` 控制（默认 `./config`）。

| 概念 | 说明 |
|------|------|
| Collector | 采集作业(如 Soundwave)。只负责爬取和投递,**不是 agent**,不做分级/推送。 |
| 数据源(Source) | 登记在 Megatron 里的源身份,由 `config/sources/*.yaml` 定义。有 `adapter` 决定数据怎么进来。 |
| 采集项(Item) | 入库的原始条目,按**采集日**(`collect_date`)归档,`(source_id, external_id)` 去重。 |
| 日包(day_bundle) | 一次日任务的完整分级产物:每条一个 tier,门铃只发其中最紧要的几条。 |
| 提示词模板(Prompt) | 决定 LLM 如何分析,可配结构化输出 schema。 |
| LLM 提供商(Provider) | 模型接入配置(`model` / `api_key` / `api_base` / 温度 / `max_tokens`)。 |
| Webhook 渠道(Channel) | 分析结果的推送目标(如钉钉)。 |
| 任务 / 模块(Task) | 把 `数据源 + 时间窗口 + 过滤 + 提示词 + 模型 + 渠道 + 定时` 组合成一条可运行的流水线。 |
| 运行(Run) | 一次任务执行,状态 `completed` / `failed`,可查看输入条数、耗时、token / 成本、投递结果。 |

## 2. 启动与登录

```bash
megatron init                    # 初始化数据库表
megatron serve dev  --port 8000  # 开发服务器(支持 --reload)
megatron serve prod --port 8000  # 生产服务器
# 或使用 Docker
docker compose up -d
```

其他常用 CLI:

- `megatron seed` — 写入默认提示词模板 + 默认用户 `admin / admin`(仅测试用,请尽快改密)。
- `megatron createsuperuser` — 交互式创建 / 更新登录用户。

访问 <http://localhost:8000> → 跳转 `/ui/login`。

**首次管理员密码**:未设 `MEGATRON_ADMIN_PASSWORD` 时,首启会随机生成一个强密码并**以 WARNING 级别打印到启动日志**(事件 `bootstrap.admin_user_created_with_generated_password`,含明文密码),用户名 `admin`,显示名 `Administrator`。生产环境建议直接用环境变量固定密码。

## 3. 推荐配置顺序

登录后按此顺序一次打通链路:

1. **LLM 提供商** `/ui/providers` — 填 `model` + `api_key`(配 `MEGATRON_MASTER_KEY` 后加密存储)。
2. **提示词模板** `/ui/prompts` — 写分析指令,可配 output schema。
3. **Webhook 渠道** `/ui/channels` — 填 webhook,可「测试」发送一条测试消息。
4. **数据源** `/ui/data/sources` — 见第 4 节。
5. **任务** `/ui/tasks` — 见第 5 节。
6. **运行与查看** `/ui/runs`。

主要页面路由:仪表盘 `/ui/dashboard`、采集数据 `/ui/data/collected`、分析结果 `/ui/data/analyzed`、任务 `/ui/tasks`、运行记录 `/ui/runs`、数据源 `/ui/data/sources`、提示词 `/ui/prompts`、提供商 `/ui/providers`、渠道 `/ui/channels`、定时 `/ui/schedules`。

## 4. 源接入

**源由 YAML 定义,不在 UI 里增删。** `sources/*.yaml` 是唯一真相,启动时(或
`megatron sources sync`)投影进数据库;UI 对这些源**只读**。这样源配置进 Git、可 review、可回滚,
也不会出现"UI 改了一半、collector 还在往老 source_id 推"的漂移。

```bash
megatron sources list                       # 看有哪些源
megatron sources validate                   # 只校验 YAML,不碰数据库
megatron sources sync                       # 投影进数据库
megatron sources pull <source_id>           # 立刻拉一次(bundle_pull / http_pull / git_pull)
```

### 源接入三角

| adapter | 数据怎么进来 | 什么时候用 |
|---|---|---|
| **`http_push`**(主) | Collector 主动 `POST /api/ingest/{source_id}` | 你能改 collector,想要最实时 |
| **`bundle_pull`** | Megatron 按 cron 去拉对方发布的**统一信封日包** | 对方已经产出符合契约的 bundle(如 Soundwave) |
| **`http_pull`** | Megatron 按 cron 拉任意 JSON / RSS,用字段映射转成 Item | 任何公开 API / 订阅源,**零代码** |
| **`git_pull`** | clone 仓库读文件(兼容路径) | 老的 Soundwave `data/` 布局 |
| **`mcp_query`** | MCP 协议 | **仅用于交互式查询历史情报,不是日更路径** |

### A. 内置 Soundwave 源(开箱即用)

`sources/twitter_security_list.yaml`,`adapter: bundle_pull`。

Soundwave 每天把日包提交到自己仓库的 `bundles/YYYY-MM-DD.json` ——
而那个文件**本身就是** `schema_version: 1` 的统一信封,所以不需要任何字段映射,拉下来直接入库。
Megatron 按 cron 去 `bundles/index.json` 看有没有新日期,有就取,并校验 `sha256`
(index 和日包是 CDN 上两个独立对象,可能读到不一致的版本 —— 对不上就跳过、等下次,
而不是吃进一个残缺的日包)。

**Soundwave 一行都不用改。**

### B. 接一个新源:写一个 YAML

```yaml
# sources/hn_frontpage.yaml
source_id: hn_frontpage          # 小写 slug,同时也是 items.source 的值
display_name: Hacker News 前页
adapter: http_pull
audience: [personal]
enabled: true

schedule:
  cron: "0 6 * * *"              # 我们去拉的时间

schedule_expect:                 # 期望到达时间,用于「今日源到达」健康检查
  timezone: Asia/Shanghai
  collect_by: "14:30"
  sla_minutes: 60

fetch:
  format: json                   # json | rss
  url: https://hn.algolia.com/api/v1/search?tags=front_page
  headers:
    Authorization: "Bearer ${HN_TOKEN}"   # ${VAR} 从环境变量取,密钥不进文件

map:                             # RSS 可省略(有默认映射)
  items: $.hits                  # 相对响应根,必须是个列表
  external_id: $.objectID        # 以下相对单条
  title: $.title
  url: $.url
  published_at: $.created_at
  metrics:
    like_count: $.points
```

路径表达式子集:`$`(自身)· `$.a.b` · `$.a[0]` · `$.a[*].b`(摊平)· 不以 `$` 开头则当常量。

### C. 让 Collector 推过来(http_push)

把 YAML 里的 `adapter` 改成 `http_push`,删掉 `schedule` / `fetch` / `map`,然后:

```bash
curl -fsS -X POST "https://<host>/api/ingest/twitter_security_list" \
  -H "Authorization: Bearer $MEGATRON_INGEST_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "schema_version": 1,
    "collect_date": "2026-07-12",
    "producer": {"name": "soundwave", "version": "1.0.0", "run_id": "gha-123"},
    "items": [
      {"external_id": "1748402774835134821",
       "content": "…", "url": "https://x.com/…", "author": "0x534c",
       "published_at": "2026-07-12T08:00:00Z",
       "metrics": {"like_count": 3}, "flags": {"is_retweet": false}}
    ]
  }'
```

`GET /api/admin/sources/{source_id}/curl` 会直接吐出这段可粘贴的 curl。

规则:

- **必须先注册源**。没在 YAML 里声明的 `source_id` 会被 **404 拒收** ——
  一个拼错的 source_id 悄悄建出第二个没人分析的源,比让 collector 报错更糟。
- 幂等:去重键 `(source_id, external_id)`,重推只增 `duplicated`。
- `collect_date` 可省(默认 UTC 今天);`schema_version` 不支持会 400 并列出支持的版本。
- Collector **不要**填 `tier` / `why_for_me` —— 那是分析层的输出。

### D. 今日源到达

`GET /api/admin/sources/arrivals` → 每个源 `ok` / `late` / `missing` / `pending` / `disabled`。

日任务开跑前会检查:某个源今天没数据 → 写进 run 的 `warnings`,**但照常出刊**。
一个 collector 挂了不该让你整天没简报。想要严格失败,在任务的 filter_config 里设
`fail_on_missing_source: true`。

## 5. 任务

- **数据源**:下拉包含已注册的源 + 现有任务用过的源(打开旧任务不会丢选)。
- **数据源引用(source_ref)**:可选二级过滤(如某个 list_id),留空 = 全部。
- **时间窗口(time_mode)**:
  - `today` — 今天(UTC)采集的;
  - `date` — 指定某天(`target_date`);
  - `range` — 日期区间;
  - `rolling` — 最近 N 小时(`window_hours`,按发布时间)。
  > **分析层只读数据库**。数据由 collector 推进来、或调度器按 cron 拉进来,
  > 任务运行时不再联网取数(`mcp_query` + `MEGATRON_MCP_LIVE_FETCH` 的兜底除外)。
- **过滤器 / max_items**:可选,控制送入 LLM 的条目数量。
- **提示词模板 / LLM 提供商 / Webhook 渠道**:选前面配置好的。
- **定时(cron)**:如 `0 7 * * *`(每天 07:00 UTC);留空 = 仅手动运行。
- **门铃 + 日刊**:见第 8 节(`megatron use-day-bundle`)。
- **多源汇合**:filter_config 里 `sources: [a, b]` 可让一个任务读多个源(默认只读它自己那个)。

## 6. 运行结果如何判断

- **completed(有内容)**:选数据 → 分析 → 投递走完。
- **completed / 无数据**:时间窗口内库里确实没数据(正常空跑)。若是**源没到达**导致的,
  run 的 `warnings` 里会写明是哪个源没来 —— 这就是"collector 挂了"和"今天确实没料"的区别。
- **failed**:分析出错。错误记录在 run 的 `error` 字段,不会伪装成"无数据"的假成功。
- **投递**:某个渠道失败**不影响** completed 状态,明细在投递日志(DeliveryLog)。

## 7. 增量与去重机制

- 去重键 **`(source_id, external_id)`**(库里即 `(items.source, items.item_id)`)。
  重复推送 / 重复拉取只增 `duplicated`,不会产生重复行。
- 每个拉取型源维护一个 **watermark**(最后成功采集的日期),只拉 watermark 之后的新日期;
  冷启动无 watermark 时全量拉(受 `max_days` 限制),之后转增量。
- 注意:一条已存在的 item 被再次推送时**保留原行**(包括原来的 `collect_date`)。
  这是幂等的正确行为 —— 但也意味着早期用别的口径灌进来的历史数据,`collect_date` 会是老口径。

## 8. 门铃与日刊

一次日任务产出一个 **day_bundle**:每条一个 tier。

| tier | 含义 | 进推送? |
|---|---|---|
| `must_see_push` | 今天必须打断你 | **是**,且总数 ≤ cap(默认 3) |
| `must_see_page` | 重要,但不值得打断 | 否 |
| `recommend` | 值得读 | 否 |
| `skim` | 扫一眼 | 否 |
| `drop` | 不该占用你时间 | 否(不进日刊) |

- **门铃**(webhook):最多 3 条、≤1200 字。只回答"要不要现在停下手上的事"。
- **日刊页**:`/day/{date}?k=<MEGATRON_DAY_TOKEN>` —— 全部条目分档展示,带「为什么给你」。
  不用登录(手机上从聊天里直接点开),但 URL 带不可猜 token;token 错 → **404**(而不是 403,
  403 会泄露那天有没有日刊)。

**上限是服务端强制的,不靠 prompt 自觉。** 模型就算报了 8 条 `must_see_push`,
也只会推 3 条,其余**降级**到日刊页(不会丢)。模型自报的 `push_item_ids` 只当排序参考。
模型编造的、对不上原文的条目会被直接丢弃并计数。

开启:任务用 `daily_intel_v1` 提示词,然后

```bash
megatron use-day-bundle --module <任务名>     # 幂等,打印 before/after
megatron use-day-bundle --module <任务名> --revert   # 切回去
```

## 9. 环境变量清单

> **全部必须带 `MEGATRON_` 前缀** —— 没有前缀的变量会被**静默忽略**。
> 首启会自动生成并持久化 `session_secret` / `admin_token` / `ingest_token` / `day_token`
> 到 `/app/data/` 下的同名点文件。

| 变量 | 用途 |
|------|------|
| `MEGATRON_ADMIN_PASSWORD` | 管理员密码(不设则首启随机生成并打印日志) |
| `MEGATRON_ADMIN_TOKEN` | 管理 API Token(生产必改) |
| `MEGATRON_SESSION_SECRET` | 会话签名密钥(生产必改) |
| `MEGATRON_MASTER_KEY` | 密钥加密主密钥(加密 `api_key` 等;不设则兼容明文) |
| `MEGATRON_INGEST_TOKEN` | 采集源推送用的 Bearer token |
| `MEGATRON_DAY_TOKEN` | 日刊页 `/day/{date}?k=…` 的不可猜 key |
| `MEGATRON_BASE_URL` | **必填**。读者访问这个实例的地址,如 `https://megatron.example.com`。推送末尾的「查看今日详情」链接是用它拼的 —— 填 `localhost`/`127.0.0.1` 的话,推到手机上那条链接点开就是死的。**`MEGATRON_ENV=prod` 时如果是回环地址,服务直接拒绝启动** |
| `MEGATRON_DATABASE_URL` | 数据库连接串。**SQLite 绝对路径要四条斜杠**:`sqlite+aiosqlite:////app/data/megatron.db` |
| `MEGATRON_CONFIG_DIR` | 产品 profile 目录(默认 `./config`,含 sources/prompts/tasks/policy.yaml) |
| `MEGATRON_MCP_LIVE_FETCH` | `off`(推荐) / `backfill`(默认) / `always`。见下 |
| `MEGATRON_INGEST_AUTO_REGISTER` | 未注册的 source_id 推进来时,建一条 **disabled** 的源(仍然拒收)。默认 `false` |
| `MEGATRON_ENV` | 运行环境(`development` / `production`) |

### MEGATRON_MCP_LIVE_FETCH

历史包袱开关。以前日任务会在运行时通过 MCP 从 GitHub 实时拉全天推文再分析 ——
这是错的(分析层应该只读数据库),但它曾经是唯一能工作的取数路径。

| 值 | 行为 |
|---|---|
| `off` | `mcp_query` 源只用于查询,日任务永不联网拉数。**内置源已改用 `bundle_pull`,可以直接设 `off`。** |
| `backfill`(默认) | 只有当天库里 0 条时,才用 MCP 补一次库,然后**照常从库读**。过渡期兜底。 |
| `always` | 旧行为。紧急回滚阀,不要长期用。 |

## 10. 部署注意 ⚠️

**如果你的容器是在这次改动之前部署的,升级前必须先抢救数据库。**

以前 `docker-compose.yml` 里写的是 `DATABASE_URL`(没有 `MEGATRON_` 前缀)——它**从来没生效过**。
数据库实际落在 `/app/megatron.db`(WORKDIR),而挂载卷是 `/app/data`。
也就是说:**那个库一直不在卷上,容器重建就没了**。

修好前缀之后,应用会去开 `/app/data/megatron.db`(卷上,当前是空的)。
不抢救就直接升级,看起来就是"数据全没了":

```bash
# 1. 趁旧容器还活着,把库备份到卷上
#    用 .backup 而不是 cp:未 checkpoint 的 -wal 里有近期写入,裸 cp 会丢
docker compose exec web sqlite3 /app/megatron.db ".backup '/app/data/megatron.db'"

# 2. 确认文件非空
docker compose exec web ls -la /app/data/

# 3. 再升级
bash deploy.sh update

# 4. 验证
docker compose exec web sqlite3 /app/data/megatron.db \
  "select source, count(*) from items group by source;"
```

同一个前缀问题也让 `MEGATRON_INGEST_TOKEN` 一直没生效 ——
ingest 接口此前实际用的是默认的 `dev-ingest-token-change-me`。升级后它会真正生效。

## 11. 常见问题排错

- **任务跑出"无数据"**:先看 run 的 `warnings` —— 如果写着某个源 `missing`,那是 collector 没到达,
  不是没料。再确认 `/ui/data/collected` 里有数据、且任务时间窗口覆盖了它们的采集日。
  手动补一次:`megatron sources pull <source_id>`。
- **推送里的日刊链接点不开**:`MEGATRON_BASE_URL` 得是**对外可访问的绝对地址**。
  开发机上跑会在 run 的 `warnings` 里写一条 `base_url_not_reachable`;
  生产(`MEGATRON_ENV=prod`)干脆起不来 —— 宁可起不来,也不要连发一周死链没人发现。
- **日刊页 404**:`?k=` 的值要等于 `MEGATRON_DAY_TOKEN`。token 错 / 不带 → 一律 404
  (故意的:403 会泄露那天有没有日刊)。
- **Collector 推过来 404 `unknown_source`**:那个 `source_id` 没在 `sources/*.yaml` 里声明。
  加上,然后 `megatron sources sync`。
- **Collector 推过来 401 / 403**:401 = 没带 `Authorization: Bearer`;403 = token 不对。
  token 看 `/app/data/.ingest_token`(首启自动生成)或你自己设的 `MEGATRON_INGEST_TOKEN`。
- **改了 YAML 但没生效**:`megatron sources sync`(或重启,或 `POST /api/admin/sources/reload`)。
  `megatron sources validate` 能在不碰数据库的情况下先校验一遍。
- **环境变量像是没读到**:检查有没有 `MEGATRON_` 前缀。没前缀的会被**静默忽略**。
- **MCP 连接测试**:仍然可用(`/ui/data/sources`),但 MCP 现在只是**查询辅路径**,不是日更路径。
